# 02 — Реактивность изменений

> Каждое изменение настроек должно применяться реактивно — как на фронте, так и на бэкенде.

## Почему

Пользователь изменил тему — он ожидает увидеть результат мгновенно, без перезагрузки. Пользователь сменил модель — следующий промпт должен уйти в новую модель. Пользователь изменил config dir — SDK должен переподключиться автоматически.

Не-реактивная система заставляет пользователя помнить: "после этого изменения нужно перезагрузить", "это применится только к новой сессии". Это cognitive load, который мы обязаны убрать.

## Правила

1. **Изменение → БД → событие → все подписчики обновляются**. Один поток данных, без ручных синхронизаций
2. **Бэкенд реагирует на изменения настроек** через EventEmitter или аналог. Смена модели → `claude.setModel()`. Смена config dir → пересоздание instance
3. **Фронтенд реагирует через StateManager**. Изменение в сторе → автоматический ре-рендер подписанных компонентов
4. **Нет "Apply" кнопок** для тривиальных настроек. Изменил → применилось. Исключение: деструктивные операции (смена config dir требует reconnect)
5. **Optimistic updates** для UI. Показываем результат сразу, откатываем при ошибке

## Поток данных

```
User action
  → StateManager.dispatch(action)
    → Middleware: persist to DB
    → Middleware: notify backend via IPC
    → State update
      → Subscribers re-render
      → Backend applies change
```

## Хорошо

```typescript
// Реактивная смена темы — мгновенно
function setTheme(theme: Theme) {
  store.dispatch({ type: 'settings/setTheme', payload: theme });
  // Middleware автоматически: 1) сохранит в БД, 2) обновит CSS-переменные
}
```

```typescript
// Бэкенд реактивно реагирует на смену модели
settingsEmitter.on('change:model', async (model) => {
  await claude.setModel(model);
});
```

## Плохо

```typescript
// Ручная синхронизация — легко забыть один из шагов
function saveSettings(s: Settings) {
  fs.writeFileSync('settings.json', JSON.stringify(s)); // 1. файл
  setSettings(s);                                        // 2. state
  ipcRenderer.send('settings:update', s);                // 3. бэкенд
  document.body.className = s.theme;                     // 4. DOM
  // Что если 3 упадёт? Фронт и бэк рассинхронизированы
}
```

## Ограничения

- **Запрещено** менять состояние в обход StateManager
- **Запрещено** требовать перезагрузку для применения настроек (кроме фатальных: смена БД, смена executable)
- **Запрещено** дублировать логику применения изменений — один handler на каждый тип изменения
- **Обязательно** деструктивные изменения (reconnect SDK) — с подтверждением пользователя
