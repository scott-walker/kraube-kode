# 01 — Персистентное состояние

> Всё состояние приложения должно быть максимально персистентным. Мы храним всё в базе данных.

## Почему

Потеря состояния — это потеря доверия пользователя. Если пользователь настроил приложение, провёл сессию, изменил тему — и после перезапуска всё сбросилось, приложение воспринимается как ненадёжное. Персистентность — это контракт с пользователем: "мы помним всё, что ты сделал".

Второй аргумент — детерминированность. Если состояние живёт только в памяти, при краше оно теряется безвозвратно. БД даёт нам recovery point — мы всегда можем восстановить последнее консистентное состояние.

## Правила

1. **Каждый фрагмент состояния имеет owner** — конкретную таблицу/коллекцию в БД
2. **In-memory state = кэш**, не источник истины. Если данные есть в памяти — они были прочитаны из БД
3. **Запись в БД — синхронная с точки зрения логики**. Пользователь не должен видеть состояние, которое ещё не персистировано
4. **Миграции обязательны**. Схема БД версионируется. Нет "просто добавим поле"
5. **Никакого localStorage/sessionStorage для бизнес-данных**. Только БД через IPC

## Что хранится в БД

| Данные | Таблица | Пример |
|--------|---------|--------|
| Настройки приложения | `settings` | executable path, config dir, theme |
| Сессии | `sessions` | session ID, summary, timestamps |
| История сообщений | `messages` | role, content, blocks, tokens |
| MCP серверы | `mcp_servers` | name, config, enabled |
| Пользовательские темы | `themes` | name, variables |

## Хорошо

```typescript
// Настройка сохраняется в БД, затем применяется
async function updateSetting(key: string, value: unknown): Promise<void> {
  await db.settings.put(key, value);
  stateManager.invalidate('settings');
}
```

## Плохо

```typescript
// Состояние живёт только в useState — потеряется при перезагрузке
const [model, setModel] = useState('sonnet');

// JSON-файл вместо БД — нет транзакций, нет миграций, гонка при параллельной записи
fs.writeFileSync('settings.json', JSON.stringify(settings));
```

## Ограничения

- **Запрещено** хранить бизнес-состояние в файлах (JSON, YAML). Только БД
- **Запрещено** полагаться на in-memory state как на источник истины
- **Запрещено** менять схему БД без миграции
- **Исключение**: кэши и транзиентное UI-состояние (позиция скролла, hover, фокус) — не персистируются
