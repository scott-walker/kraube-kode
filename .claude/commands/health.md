# System Health Check

Ты — архитектурный аудитор проекта Kraube Kode. Твоя задача — оценить текущее состояние кодовой базы на соответствие 12 архитектурным принципам, задокументированным в `docs/architecture/`.

## Инструкция

1. Прочитай все файлы в `docs/architecture/` (01–12) чтобы понимать критерии
2. Проанализируй текущий код в `src/` — каждый `.ts`, `.tsx`, `.css` файл
3. Для каждого из 12 принципов выстави оценку и дай краткий вердикт

- 🔴 Критично (0-3) — принцип не соблюдён, срочный рефакторинг

## Критерии оценки

### 01 — Персистентность
- 10: Все данные в БД, миграции, нет JSON-файлов для бизнес-данных
- 5: Часть данных персистирована, часть в памяти
- 0: Состояние только в useState/переменных, теряется при перезагрузке

### 02 — Реактивность
- 10: Изменение → автоматическое обновление везде, нет ручных синхронизаций
- 5: Часть реактивна, часть требует ручного обновления
- 0: Применение настроек требует перезагрузки

### 03 — StateManager
- 10: Централизованный стор, computed properties, actions, middleware
- 5: Частичная централизация, часть данных в useState
- 0: Весь state в разрозненных useState по компонентам

### 04 — Контракты
- 10: Все внешние зависимости за интерфейсами, DI, порты/адаптеры
- 5: Частичная абстракция, некоторые прямые импорты
- 0: Прямые зависимости от реализаций повсюду

### 05 — Расширяемость
- 10: Registry pattern, plugin API, точки расширения
- 5: Некоторые реестры, но есть хардкод
- 0: Всё захардкожено, switch/case для типов

### 06 — Семантика
- 10: Единый глоссарий, консистентные имена во всём стеке
- 5: В основном ок, но есть расхождения между слоями
- 0: Хаотичные имена, одна сущность — разные названия

### 07 — Декомпозиция
- 10: Все файлы < 100 строк, чёткая структура директорий по доменам
- 5: Большинство < 150 строк, есть God-файлы
- 0: Файлы > 300 строк, всё в одной директории

### 08 — Безопасность бэкенда
- 10: Cleanup везде, mutex, AbortController, нет утечек
- 5: Базовый cleanup, но есть потенциальные утечки
- 0: Listeners без cleanup, fire-and-forget, гонки

### 09 — SSOT
- 10: Каждый факт в одном месте, нет дублирования типов/логики
- 5: Частичное дублирование, некоторые computed хранятся как state
- 0: Массовое дублирование данных и логики

### 10 — Чистый фронтенд
- 10: Компоненты = рендеринг, логика в хуках/сервисах, < 100 строк
- 5: Часть компонентов чистые, часть содержат логику
- 0: Бизнес-логика, API-вызовы и рендеринг в одних компонентах

### 11 — Дизайн-система
- 10: Ноль inline-стилей, все значения через токены, темы через переменные
- 5: Часть через CSS, часть inline
- 0: Всё inline, хардкод цветов и размеров

### 12 — Перформанс
- 10: memo, гранулярные подписки, виртуализация, < 16ms/frame
- 5: Базовая оптимизация, но есть лишние ре-рендеры
- 0: Полные перерисовки на каждый чанк стрима
