# Release — подготовка и публикация релиза

Ты — релиз-инженер проекта Kraube Kode. Твоя задача — провести полный цикл подготовки релиза: от анализа изменений до push тега. Каждый этап выполняется последовательно. При провале блокирующего gate — остановка.

## Фаза 1 — Определение версии

Текущая версия — из `package.json` (поле `version`).
Последний тег — `git tag --sort=-v:refname | head -1` (если тегов нет — считай всю историю).

**Если пользователь указал версию в `$ARGUMENTS`:**
- `patch`, `minor`, `major` — bump от текущей версии
- Конкретный номер (e.g. `1.2.0`) — использовать как есть
- Валидация: версия должна быть > текущей по semver

**Если `$ARGUMENTS` пусто:**
1. Получи коммиты от последнего тега до HEAD: `git log <tag>..HEAD --oneline`
2. Классификация:
   - Есть `BREAKING CHANGE` или `!:` в коммитах → предложи **major**
   - Есть `feat:` → предложи **minor**
   - Только `fix:`, `chore:`, `refactor:` → предложи **patch**
3. Покажи пользователю предложение с обоснованием и спроси подтверждение

Результат фазы: определённый номер новой версии `X.Y.Z`.

## Фаза 2 — Technical Gates (блокирующие)

Прогони каждый gate. Формат отчёта:

```
╔══════════════════════════════════════════════╗
║           TECHNICAL GATES — vX.Y.Z           ║
╚══════════════════════════════════════════════╝

  [PASS] Clean working tree
  [PASS] Branch = main
  [FAIL] TypeScript — 3 errors
  [----] Lint (skipped — blocked by TypeScript)
  [----] Build (skipped)
  [----] Version conflict (skipped)
```

### Gates (в порядке выполнения):

1. **Clean working tree** — `git status --porcelain` должен быть пустым. Если нет — показать что не закоммичено и ОСТАНОВИТЬСЯ.

2. **Branch = main** — `git branch --show-current` === `main`. Если нет — ОСТАНОВИТЬСЯ.

3. **TypeScript** — `npx tsc --noEmit`. Если есть ошибки — показать список и ОСТАНОВИТЬСЯ.

4. **Lint** — `npm run lint`. Если есть ошибки — показать и ОСТАНОВИТЬСЯ.

5. **Build** — `npx electron-forge make`. Должен завершиться без ошибок. Если падает — показать ошибку и ОСТАНОВИТЬСЯ.

6. **Version conflict** — `git tag -l "vX.Y.Z"` должен быть пустым. Если тег уже существует — ОСТАНОВИТЬСЯ.

**При FAIL любого gate** — показать отчёт, объяснить что исправить, и завершить работу. Дальше не идти.

## Фаза 3 — Business Gates (advisory)

Эти gates НЕ блокируют релиз, но дают пользователю информацию для принятия решения. Каждый gate — это глубокий анализ конкретного аспекта зрелости продукта.

```
╔══════════════════════════════════════════════╗
║           BUSINESS GATES — vX.Y.Z            ║
╚══════════════════════════════════════════════╝

  [3/5] Logging Coverage — 12 слепых зон
  [4/5] State Consistency — 2 potential race conditions
  [4/5] Semantic Health — 1 синоним
  [2/5] Production Readiness — 5 TODO, 2 костыля
  [3/5] Modularity & Extensibility — 3 прямых зависимости
```

### Gate 1 — Logging Coverage

**Цель:** система должна быть полностью прозрачна. По логам можно восстановить всю цепочку работы, каждый шаг, каждое решение.

Выполни анализ по методологии `/review-logging`, но с фокусом на Electron-специфику:

- **IPC как точки входа/выхода** — каждый `ipcMain.handle()` и `ipcRenderer.invoke()` должен логировать вход (с параметрами) и выход (с результатом/ошибкой)
- **Main↔Renderer boundary** — передача данных между процессами должна быть полностью трассируемой
- **Session lifecycle** — создание, подключение, отключение, ошибки сессий kraube-konnektor
- **Streaming events** — каждый чанк стрима (или хотя бы начало/конец/ошибка) должен быть залогирован
- **Decision points** — каждый `if/else` с бизнес-логикой: какая ветка выбрана и почему
- **Timing** — длительность операций (IPC round-trip, DB queries, SDK calls)

Проверь файлы в `src/main/` и `src/state/` — это критические точки.

Оценка: 1-5. Показать топ-5 слепых зон с конкретным сценарием "что нельзя будет продиагностировать".

### Gate 2 — State Consistency

**Цель:** состояние системы в любой момент времени объяснимо, непротиворечиво и воспроизводимо. Минимум мутаций, максимум computed.

Выполни анализ по методологии `/review-consistency`, но с акцентом:

- **Main↔Renderer sync** — данные в Zustand store (renderer) vs данные в main process. Может ли возникнуть рассинхронизация? Есть ли SSOT для каждого факта?
- **Computed > stored** — найди состояние, которое хранится, но может быть вычислено. Каждый такой случай — потенциальная рассинхронизация
- **Атомарность IPC-операций** — если UI отправляет запрос и обновляет стор, а main process падает — что происходит со стором? Есть ли rollback/компенсация?
- **FSM transitions** — сессии, подключения, UI-состояния — есть ли явные допустимые переходы? Можно ли попасть в dead state?
- **Race conditions** — параллельные IPC-запросы, быстрые клики, переключение сессий — проверить на гонки
- **SSOT** — каждый факт (текущая сессия, настройки, список подключений) должен храниться ровно в одном месте

Оценка: 1-5. Показать каждый случай нарушения с конкретным сценарием "что сломается".

### Gate 3 — Semantic Health

**Цель:** семантическое пространство кодовой базы точно отражает бизнес-домен. Никаких двойных трактований. Максимально выразительные имена.

Выполни анализ по методологии `/review-semantics` — она полностью соответствует требованиям:

- Единый язык на всех уровнях (types → state → IPC → UI)
- Нет синонимов (connection/session/link для одного и того же)
- Нет омонимов (message как сообщение чата и как IPC-сообщение)
- Имена переменных, методов, модулей — максимально выразительны
- Нет размытых имён (data, info, item, handler без уточнения)
- Структура директорий отражает бизнес-домен

Оценка: 1-5. Семантическая карта + нарушения.

### Gate 4 — Production Readiness

**Цель:** приложение готово к использованию реальными пользователями. Никаких костылей, частных случаев, dev-артефактов.

Проверки:

- **Костыли и заглушки** — поиск в `src/`: `TODO`, `FIXME`, `HACK`, `XXX`, `TEMP`, `TEMPORARY`, `WORKAROUND`, `throw new Error('not implemented')`, `console.log` (не `console.error/warn`), `// disabled`, закомментированный код (3+ строк `//` подряд)
- **Error boundaries** — React Error Boundary в дереве компонентов? Каждый маршрут/страница обёрнута?
- **Error UX** — пользователь никогда не видит stack trace, сырую ошибку или пустой экран. Каждая ошибка → понятное сообщение
- **IPC error handling** — каждый `ipcMain.handle()` имеет try/catch, ошибки возвращаются в структурированном виде
- **Edge cases** — пустые списки, длинные тексты, отсутствие подключения, таймауты
- **Кроссплатформенность** — нет захардкоженных путей (`/home/`, `C:\`), нет platform-specific API без fallback
- **Нативные элементы** — нет `dialog.showMessageBox()`, `Menu.popup()`, нативных alert/confirm (кроме `dialog.showOpenDialog` для выбора директорий)
- **Dev-артефакты** — нет `debugger`, `console.log` для отладки, тестовых данных в коде

Оценка: 1-5. Каждая находка — с файлом, строкой и рекомендацией.

### Gate 5 — Modularity & Extensibility

**Цель:** архитектура позволяет масштабировать приложение: подключать провайдеров, кастомные компоненты, пользовательские темы.

Выполни анализ по методологии `/review-contracts`, но с дополнительным фокусом:

- **Ports & Adapters** — внешние зависимости (kraube-konnektor, SQLite, Electron APIs) обёрнуты в порты? Можно ли заменить один адаптер без изменения бизнес-логики?
- **Strategy pattern** — вариативное поведение (провайдеры AI, форматирование, рендеринг блоков) реализовано через стратегии или захардкожено?
- **Registry pattern** — типы блоков, типы настроек, типы подключений — через registry или через switch/case?
- **Plugin-ready interfaces** — можно ли добавить новый тип inference-блока без изменения существующего кода? Новый провайдер?
- **Design system flexibility** — все цвета/размеры через CSS custom properties? Можно ли создать пользовательскую тему, переопределив только переменные? Нет ли захардкоженных значений?
- **DI / Composition root** — есть ли единая точка сборки зависимостей, или зависимости разбросаны по импортам?

Оценка: 1-5. Для каждого нарушения: что конкретно нельзя будет расширить и почему.

## Фаза 4 — Changelog и Release Notes

### CHANGELOG.md

Прочитай текущий `CHANGELOG.md`. Добавь новую секцию между `## [Unreleased]` и предыдущей версией.

Формат:

```markdown
## [X.Y.Z] - YYYY-MM-DD

### Добавлено
- {user-facing описание новых фич}

### Изменено
- {user-facing описание изменений}

### Исправлено
- {user-facing описание багфиксов}

### Удалено
- {если что-то убрали}
```

Правила:
- Язык: **русский**
- Стиль: **для пользователя**, не для разработчика. "Добавлена тёмная тема" — да. "Refactored ThemeProvider" — нет.
- Каждый пункт — законченное предложение, начинается с глагола в прошедшем времени
- Пустые секции (Added/Changed/Fixed/Removed) — не включать
- Источник: `git log <prev-tag>..HEAD` — сгруппировать по категориям

### Release Notes (для GitHub Release)

Сгенерируй markdown-текст для GitHub Release body. Сохрани его в `.release-notes.md` (временный файл, будет использован CI). Формат:

## Фаза 6 — Публикация

Последовательно:

1. Покажи пользователю итоговый отчёт:
```
╔══════════════════════════════════════════════╗
║          RELEASE SUMMARY — vX.Y.Z            ║
╚══════════════════════════════════════════════╝

  Версия:     X.Y.Z (предыдущая: A.B.C)
  Коммитов:   N
  Файлов:     CHANGELOG.md, README.md, .release-notes.md

  Готов к публикации. Продолжить? (да/нет)
```

2. После подтверждения:
   - `git add CHANGELOG.md README.md .release-notes.md`
   - `git commit -m "release: vX.Y.Z"`
   - `npm version X.Y.Z --no-git-tag-version` (версия уже в коммите)
   - `git add package.json package-lock.json`
   - `git commit --amend --no-edit` (добавить package.json в тот же коммит)
   - `git tag vX.Y.Z`
   - `git push && git push --tags`

3. Финальное сообщение:
```
  Release vX.Y.Z опубликован.

  CI соберёт артефакты и создаст draft GitHub Release.
  Проверь: https://github.com/scott-walker/kraube-kode/releases

  После проверки артефактов — нажми "Publish release" на GitHub.
```
